# NextBio 영업앱 — 실데이터 연결 & 배포 가이드

> 작성 2026-06-19 · 비개발자용 설명서. "각 단계에서 무엇을 준비하고, 무엇이 위험하며, 데이터가 어떻게 흐르고 이후 어떻게 업데이트되는지"를 정리.

---

## 0. 큰 그림 — 배포는 2단계로 갑니다

| | 1단계 (지금) — **데모 배포** | 2단계 — **실데이터 배포** |
|---|---|---|
| 데이터 저장 | 각자 브라우저(localStorage) | **중앙 DB(Supabase)** — 모두 공유·영구 |
| 로그인 | 데모(비번 1234) | **진짜 구글 로그인 + 권한** |
| 데이터 | 샘플(가짜) | **실제 거래처·제품·단가** |
| 용도 | 구경·피드백·UX 확인 | 실제 업무 사용 |
| 위험도 | 낮음(가짜 데이터) | 관리 필요(아래 참고) |

**핵심 원칙: 1단계(데모)에는 실제 데이터를 절대 넣지 마세요.** 저장 안전성·보안이 없습니다. 실데이터는 2단계부터.

---

## 1. 단계별 — 준비물 / 위험 / 대비

### 1단계: 데모를 Vercel에 배포 (지금)
- **준비물**: Vercel 계정(이미 있음) → 로그인 1회.
- **되는 것**: 링크 하나로 폰·PC에서 열림(항상 켜짐). 직원에게 "이렇게 생겼다" 보여주기 가능.
- **위험**
  - 데이터가 **각자 브라우저에만** 저장됨 → 다른 사람과 공유 안 됨, 초기화하면 사라짐. (데모라 정상)
  - 링크를 아는 사람은 **누구나 접속 + 데모 계정 로그인** 가능 (진짜 보안 아님).
- **대비**: 링크는 **내부에만** 공유. 실데이터 입력 금지. "미리보기"라고 명확히 공지.

### 2단계: 중앙 DB 연결 (Supabase)
- **준비물**: Supabase 무료 계정 + 프로젝트 1개 → **Project URL** + **anon key** 2개.
- **위험**
  - 🔑 **키 2종 구분**: `anon key`(공개용 — 앱에 넣어도 안전) vs `service_role key`(**비밀** — 절대 코드/깃에 넣지 말 것). 혼동 시 전체 데이터 노출 위험.
  - 🔓 **권한(RLS) 미설정**: 기본 상태면 로그인한 누구나 모든 팀 데이터를 볼 수도 있음.
- **대비**: service key는 절대 커밋 안 함(이미 `.gitignore`에 `.env` 제외돼 있음). **RLS(행 수준 보안)** 로 "영업1팀 계정은 1팀 데이터만" DB 차원에서 강제 → 영업1팀 계정으로 2팀 데이터가 안 보이는지 직접 테스트.

### 3단계: 구글 로그인 (인증)
- **준비물**: Google Cloud OAuth 클라이언트(ID/secret) 생성 → Supabase에 등록. (제가 화면 단계별로 안내, ~10분)
- **위험**
  - 🚪 설정만 하면 **아무 구글 계정이나** 로그인될 수 있음 → 회사 사람만 들어오게 제한 필요.
  - 리디렉트 URL을 잘못 넣으면 로그인 실패.
- **대비**: **등록된 사용자(이메일 화이트리스트)만 통과** + 가능하면 회사 도메인(@nextbio…) 제한. 미등록 계정은 자동 차단.

### 4단계: 실제 데이터 이관
- **준비물**: 기존 엑셀(거래처·제품·단가·배송지) + **거래처별 팀 배정**.
- **위험**
  - 엑셀의 **품목코드 과학표기(4.0005E7)**, **중복 거래처**, **표기 흔들림**을 그대로 넣으면 오류·중복.
  - 매핑 실수(거래처↔팀, 제품↔단가).
- **대비**: 제가 **정제 후** 넣고, 넣기 전 **검증표**(건수·중복·이상치) 보여드림. **기존 시트는 그대로 보존**(원본 안 건드림), 새 DB는 별도라 문제 시 **다시 넣기(롤백)** 쉬움.

### 5단계: 운영 & 업데이트 (배포 이후)
- **데이터 흐름**: 사용자가 앱에서 입력/수정 → **Supabase에 즉시 저장** → 모든 사람·기기에 반영 + **이력** 남음.
- **기능 수정**: 화면·기능은 코드 고쳐서 **재배포(자동)** — 지금처럼 계속 개선 가능.
- **데이터 구조 변경**(예: 새 칸 추가): "DB에도 칸 추가"하는 **마이그레이션 한 단계**가 붙음(제가 처리, **기존 데이터 보존**).
- **백업**: Supabase 자동 백업(유료) 또는 수동 export. → 주기적 export 권장.
- **비용 관리**: 무료로 시작 → 사용량 늘면 알림 보고 유료 전환.

## 2. 데이터 흐름 — 지금 vs 실데이터

```
[지금/데모]   입력 → 내 브라우저(localStorage) → 나만 봄 (공유·영구 X)

[실데이터]    입력/수정 ─▶ Supabase 중앙 DB ─▶ 모든 사용자에 실시간 반영
                                  │              + 권한별 필터(팀 격리)
                                  │              + 누가·언제 수정 이력
                                  └─▶ 유상 출고 → 매출 자동 집계 → KPI 대시보드
```

**향후 연동 지점**: 지금은 영업만. 나중에 생산·품질·물류·재고 모듈도 **같은 DB**에 붙이면 수주→생산→검수→출고→재고가 자동으로 이어집니다. (영업앱을 먼저 안정화한 뒤 확장)

## 3. 위험 요약표

| 위험 | 영향 | 대비 |
|---|---|---|
| 실제 데이터를 깃/공개 저장소에 올림 | 사업정보 유출 | `erp-analysis/data` 깃 제외(완료), 비밀키 커밋 금지 |
| 데모에 실데이터 입력 | 유실·노출 | 데모=미리보기 전용, 실데이터는 2단계부터 |
| service_role 키 노출 | DB 전체 탈취 | 비밀키는 서버/환경변수만, 프론트엔 anon만 |
| RLS 미설정 | 타 팀 데이터 열람 | 팀 격리 정책 + 교차 테스트 |
| 구글 로그인 무제한 | 외부인 접속 | 허용 이메일/도메인 제한 |
| 엑셀 그대로 이관 | 코드·중복 오류 | 정제+검증표+롤백 |

## 4. 사장님 액션 체크리스트 (순서)

- [ ] **1단계**: Vercel 로그인 1회 → (제가) 데모 배포 → 링크 확인
- [ ] **2단계**: Supabase 가입·프로젝트 생성 → URL + anon key 전달
- [ ] **정보**: 사용자 명단(이름·구글이메일·팀·역할), 거래처별 팀 배정
- [ ] **3단계**: (제 안내로) 구글 OAuth 설정
- [ ] **4단계**: 실데이터 이관 검증표 확인
- [ ] **5단계**: 실데이터 링크로 팀 테스트 → 정식 사용
